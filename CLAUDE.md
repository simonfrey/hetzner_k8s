## Development rules

### Setup from 0

Make sure everything you do does not haec any prerequisites. We always "terraform destroy" and setup from 0. No state kept.

### Persistance

Whenever you change something, never apply anything in plance (e.g. via kubeapply from stdin), but rather make sure to 
persist the changes in the git repository. This way, we can always go back to a previous state if something goes wrong, 
and also have the changes documented for future reference.


## Kubernetes manifests first

Whenever kuberentes resources are creates, you MUST create a ".yaml" manifest. Never create the resources directly in the cluster.
Do NOT use any kubectl create commands, only kubectl apply with manifests stored in the git repository.

## Debugging rules

### Kubernetes debugging with MCP

You have a "kubernetes" MCP connected with read-only permissions to the cluster. You can use this MCP to debug the cluster without risking any changes. 
Always prefer this MCP for debugging over any other method that might have write permissions.

## KubeVirt / Nested Virtualization

Hetzner Cloud does NOT support nested virtualization, even on dedicated vCPU (CCX) instances. The KVM kernel modules (`kvm`, `kvm_amd`) were removed from the kubevirt_worker Talos config because they have no effect. KubeVirt runs with `useEmulation: true` (software emulation via QEMU TCG).

If migrating to a provider that supports nested virtualization, re-add `kvm`/`kvm_amd` kernel modules to the kubevirt_worker config in `talos.tf` and set `useEmulation: false` in `gitops/apps/kubevirt-cr/kubevirt-cr.yaml`.

### BIOS boot required under TCG

VMs MUST use BIOS boot (SeaBIOS), not UEFI (OVMF). OVMF cannot boot SATA CDROMs under QEMU TCG software emulation (AHCI timeouts). If migrating to a provider with KVM support, UEFI boot can be re-enabled.

### Virtio-scsi for ISO CDROMs

ISO CDROMs MUST use `bus: scsi` (virtio-scsi), not `bus: sata`. The AHCI/SATA controller under QEMU TCG causes I/O timeouts (error 0xc00000e9) on large sequential reads like `boot.wim`. Virtio-scsi is paravirtualized and bypasses AHCI entirely. SeaBIOS includes a built-in virtio-scsi driver for booting. The `vioscsi` Windows driver paths must be included in autounattend.xml so WinPE can access the CDROM after boot. Small CDROMs (virtio-drivers, sysprep) can remain on SATA.

### ISO loading via kubectl cp

Do NOT use CDI `virtctl image-upload` for ISO files. CDI converts/resizes ISOs during upload, corrupting the ISO 9660 structure. Instead, use `kubectl cp` to copy the ISO byte-for-byte into the PVC as `disk.img`.

## ArgoCD GitOps

All workloads are managed by ArgoCD using the app-of-apps pattern. The directory structure:
- `gitops/root-app/` — Root Helm chart, each template is an ArgoCD Application
- `gitops/apps/` — Individual app directories (manifests or Helm charts)
- `argocd.tf` — Terraform bootstraps Cilium, ArgoCD, secrets, and the root Application

### Adding new apps

1. Create a directory under `gitops/apps/<app-name>/` with manifests or a Helm chart
2. Add an ArgoCD Application template in `gitops/root-app/templates/<app-name>.yaml`
3. If the app needs Terraform-generated values (passwords, tokens), create secrets/configmaps in `argocd.tf` and reference them from the app manifests

### Debugging ArgoCD

- Check app sync status: `kubectl -n argocd get applications`
- ArgoCD UI: `kubectl -n argocd port-forward svc/argocd-server 8080:80`
- App details: `kubectl -n argocd get application <name> -o yaml`
- Use the kubernetes MCP for read-only debugging (see Debugging rules above)
- Sync waves control ordering — if an app fails, check if its dependencies (lower wave numbers) are healthy first

### Secrets

Passwords are generated by `random_password` in Terraform. Retrieve via:
- `terraform output -raw windows_admin_password`
- `terraform output -raw guacamole_password`
- `terraform output -raw grafana_password`

Terraform creates `kubernetes_secret` and `kubernetes_config_map` resources. ArgoCD-managed apps reference these by name — they do NOT manage the secrets themselves.
