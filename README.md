# Hetzner K8s Cluster (Talos Linux + ArgoCD GitOps)

Production-ready Kubernetes cluster on Hetzner Cloud using Talos Linux — an immutable, API-only OS purpose-built for Kubernetes. All workloads are managed by ArgoCD via GitOps.

```
Internet → *.k8s.simon-frey.com → Hetzner LB → Traefik Ingress → Your Apps
```

## Architecture

| Component | Implementation |
|-----------|---------------|
| IaC | Terraform with `hcloud`, `talos`, `helm`, `kubernetes` providers |
| GitOps | ArgoCD (app-of-apps pattern with sync waves) |
| OS | Talos Linux (immutable, no SSH) |
| Image Build | Packer (Hetzner snapshot with qemu-guest-agent) |
| CNI | Cilium (kube-proxy replacement) |
| Networking | Hetzner Private Network (10.0.0.0/16, eu-central zone) |
| Load Balancer | Hetzner Cloud LB11 (TCP passthrough + proxy protocol) |
| Ingress | Traefik (Helm) |
| TLS | cert-manager + Let's Encrypt (HTTP-01 per service) |
| Cloud integration | Hetzner CCM + CSI (Helm) |
| Auto-scaling | Cluster Autoscaler (0-5 workers) |
| VPN Access | wireproxy (userspace WireGuard, K8s + Talos API restricted to VPN) |
| Virtualization | KubeVirt + CDI (Windows Server 2022 via browser RDP) |
| Monitoring | kube-prometheus-stack (Prometheus + Grafana, optional) |

### What Terraform manages (bootstrap)

Terraform handles the chicken-and-egg resources that must exist before ArgoCD can function:

- Infrastructure: servers, network, firewall, load balancer, WireGuard tunnel
- Talos config + bootstrap
- **Cilium CNI** (nodes are NotReady without it)
- **ArgoCD** (the GitOps controller itself)
- **Root Application** (app-of-apps entry point)
- Kubernetes secrets/configmaps with Terraform-generated values (passwords, tokens)

### What ArgoCD manages (everything else)

ArgoCD syncs all remaining workloads from the `gitops/` directory, ordered by sync waves:

| Wave | Component | Type |
|------|-----------|------|
| -5 | Hetzner CCM | Upstream Helm chart |
| -4 | Hetzner CSI | Upstream Helm chart |
| -3 | metrics-server | Upstream Helm chart |
| -2 | Traefik | Upstream Helm chart |
| -1 | cert-manager | Upstream Helm chart |
| 0 | cert-issuers | Local Helm chart |
| 1 | KubeVirt + CDI operators | Local manifests |
| 2 | KubeVirt + CDI CRs | Local manifests |
| 3 | Windows VM + Guacamole | Local Helm chart |
| 4 | Cluster autoscaler | Local manifests |
| 5 | Monitoring (optional) | Upstream Helm chart + local values |

## Security Features

- **No SSH**: Talos has no shell, no SSH — managed entirely via API
- **K8s API Access**: Restricted to WireGuard tunnel via wireproxy (port 6443 not publicly exposed)
- **Talos API Access**: Restricted to WireGuard tunnel (port 50000 not publicly exposed)
- **Immutable OS**: Read-only rootfs, no package manager, minimal attack surface
- **Pod Security Standards**: Restricted policy enforced on demo namespace
- **Security Contexts**: All workloads run as non-root with dropped capabilities
- **Firewall**: Explicit allow-list on public interface (Hetzner FW does not filter private network)
- **RBAC**: Minimal permissions for cluster autoscaler
- **PV Reclaim Policy**: Retain (prevents accidental data loss)
- **Secrets in Terraform state**: Passwords never stored in git — generated by Terraform, retrievable via `terraform output`

## Quick Start

### 1. Prerequisites

```bash
# Install tools
brew install terraform kubectl helm wireproxy packer jq  # macOS
# pacman -S terraform kubectl helm packer jq && go install github.com/pufferffish/wireproxy/cmd/wireproxy@latest  # Arch

# Create WireGuard keys (server + client)
mkdir -p ~/.wireguard
wg genkey | tee ~/.wireguard/hetzner-server-private | wg pubkey > ~/.wireguard/hetzner-server-public
wg genkey | tee ~/.wireguard/hetzner-client-private | wg pubkey > ~/.wireguard/hetzner-client-public

# Create Hetzner Cloud project + API token
# → https://console.hetzner.cloud → New Project → API Tokens → Generate (Read/Write)
```

### 2. Configure

```bash
cat > terraform.tfvars << 'EOF'
hcloud_token       = "your-token-here"
letsencrypt_email  = "you@example.com"
enable_monitoring  = false    # set true for Prometheus + Grafana
enable_windows_vm  = true     # set false to skip KubeVirt Windows VM
EOF
```

### 3. Build Talos Image

Build the Talos snapshot once (or when upgrading Talos versions):

```bash
cd packer
packer init .
packer build -var "hcloud_token=$(sed -n 's/.*hcloud_token\s*=\s*"\([^"]*\)".*/\1/p' ../terraform.tfvars)" .
cd ..
```

### 4. Deploy

Terraform provisions infrastructure, bootstraps Talos, installs Cilium + ArgoCD, and ArgoCD syncs all workloads from git.

```bash
export TF_VAR_wireguard_server_private_key="$(cat ~/.wireguard/hetzner-server-private)"
export TF_VAR_wireguard_server_public_key="$(cat ~/.wireguard/hetzner-server-public)"
export TF_VAR_wireguard_client_public_key="$(cat ~/.wireguard/hetzner-client-public)"
export TF_VAR_wireguard_client_private_key="$(cat ~/.wireguard/hetzner-client-private)"

terraform init
terraform plan
terraform apply
```

Terraform writes `.wireproxy.conf` in the project root and starts wireproxy in the background (PID in `.wireproxy.pid`). It creates TCP tunnels from `127.0.0.1:6443` (K8s API) and `127.0.0.1:50000` (Talos API) over WireGuard. On `terraform destroy`, the process is stopped and config files cleaned up automatically.

### 5. Fetch Kubeconfig and Verify

```bash
./scripts/post-deploy.sh
```

This fetches the kubeconfig, rewrites the API URL for the WireGuard tunnel, waits for ArgoCD, and shows the cluster status. All workloads are deployed automatically by ArgoCD — no manual helm/kubectl commands needed.

### 6. DNS Setup

After deploy, add these records in Cloudflare (DNS-only mode, gray cloud):

| Type | Name | Value |
|------|------|-------|
| A | `*.k8s` | `<LB_IP from terraform output>` |
| A | `k8s` | `<LB_IP from terraform output>` |

### 7. Test

```bash
# Check ArgoCD applications are synced
kubectl -n argocd get applications

# Deploy example app
kubectl apply -f manifests/example-app.yaml

# Watch certificate provisioning
kubectl get certificate -n demo -w

# Test (after certificate is ready)
curl https://hello.k8s.simon-frey.com
```

### 8. Windows ISO (manual, if using Windows VM)

The Windows ISO must be copied manually into the PVC (CDI corrupts ISOs during upload):

```bash
# Place ISO in the project
cp /path/to/SERVER_EVAL_x64FRE_de-de.iso ISOs/

# Copy into PVC
./scripts/copy-windows-iso.sh
```

## Adding a New Service

Each service gets its own certificate automatically:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: traefik
  tls:
    - hosts: ["myapp.k8s.simon-frey.com"]
      secretName: myapp-tls
  rules:
    - host: myapp.k8s.simon-frey.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-app
                port:
                  number: 80
```

### Adding an ArgoCD-managed app

1. Create a directory under `gitops/apps/<app-name>/` with manifests or a Helm chart
2. Add an ArgoCD Application template in `gitops/root-app/templates/<app-name>.yaml`
3. Commit and push — ArgoCD syncs automatically

## Connecting to the Cluster

After the initial deploy, `terraform apply` starts wireproxy automatically. If you reboot your machine or stop the tunnel, reconnect like this:

```bash
# Start the wireproxy tunnel (uses the config generated by Terraform)
nohup wireproxy -c .wireproxy.conf > .wireproxy.log 2>&1 &
echo $! > .wireproxy.pid

# Verify the tunnel is up (K8s API on localhost:6443, Talos API on localhost:50000)
bash -c "echo >/dev/tcp/127.0.0.1/6443" 2>/dev/null && echo "Tunnel OK" || echo "Tunnel not ready"

# Use kubectl with the generated kubeconfig
export KUBECONFIG=$(pwd)/.kubeconfig
kubectl get nodes
```

> **Note:** Both the K8s API (`127.0.0.1:6443`) and Talos API (`127.0.0.1:50000`) are only reachable through the wireproxy WireGuard tunnel. Without it, `kubectl` cannot reach the cluster.

## Operations

```bash
# View cluster nodes
kubectl get nodes -o wide

# Check ArgoCD sync status
kubectl -n argocd get applications

# View autoscaler logs
kubectl -n cluster-autoscaler logs -f deployment/cluster-autoscaler
```

## Troubleshooting

| Issue | Check |
|-------|-------|
| Can't connect to API | Is wireproxy running? `ps -p $(cat .wireproxy.pid)` and check `.wireproxy.log` |
| Nodes NotReady | Has Cilium been installed? `kubectl -n kube-system get pods -l app.kubernetes.io/name=cilium-agent` |
| ArgoCD app stuck | `kubectl -n argocd get application <name> -o yaml` — check sync status and conditions |
| App out of sync | Check sync waves — lower-wave dependencies must be healthy first |
| Workers not joining | Check autoscaler logs: `kubectl -n cluster-autoscaler logs deploy/cluster-autoscaler` |
| LB health checks failing | `kubectl get pods -n traefik` — is Traefik running? |
| Cert not issuing | `kubectl describe certificate -n <namespace>` and `kubectl logs -n cert-manager deploy/cert-manager` |
| Pods stuck Pending | `kubectl describe pod <pod>` — check resource constraints or cloud-provider taint |
| No real client IPs | Check Traefik proxy protocol config in ArgoCD app: `kubectl -n argocd get app traefik -o yaml` |

## Learnings

### Don't use `kubernetes_manifest` — use `helm_release` with a local chart

The Terraform `kubernetes_manifest` resource connects to the Kubernetes API **during `terraform plan`** to fetch the OpenAPI schema for validation. On a fresh cluster (first `terraform apply`), the API server doesn't exist yet, so planning fails with:

```
Error: Invalid configuration for API client
  Get "https://127.0.0.1:6443/apis": dial tcp 127.0.0.1:6443: connect: connection refused
```

This is a chicken-and-egg problem: Terraform needs to plan all resources before creating any, but `kubernetes_manifest` needs the cluster to exist at plan time.

**Fix:** Use `helm_release` with a small local Helm chart instead. `helm_release` only connects to the cluster at apply time, not during planning. The root ArgoCD Application is deployed via `gitops/bootstrap/` — a tiny chart containing just the Application CR. Other `kubernetes_*` resources (`kubernetes_namespace`, `kubernetes_secret`, `kubernetes_config_map`) are fine because they also only connect at apply time.
